================================================================
  DEBUG SESSION — siege + valgrind sur /cgi-bin/echo.py
================================================================

POINT DE DEPART
---------------
Commande de test:
  siege -b -c 5 -t 2S http://127.0.0.1:8080/cgi-bin/echo.py

Symptomes observes:
  - Availability ~90% puis DEGRADATION rapide vers 45%
  - Beaucoup de reponses 500 "fantomes" (status 500 inattendu)
  - Valgrind: "Invalid read of size 8" (use-after-free)
  - Logs serveur: "epoll_ctl Bad file descriptor",
    "recv(): Socket operation on non-socket",
    "BrokenPipeError" cote CGI Python
  - Boucles de reponses avec "headers: {}" vide
  - ~50 "_remove()" consecutifs sur le meme fd


================================================================
  FIXES APPLIQUES
================================================================


[1] USE-AFTER-FREE dans Client::clear()
----------------------------------------
Fichiers: srcs/CGI/public.cpp, srcs/CGI/private.cpp

Probleme:
  Client::clear() iterait sur _cgis et faisait `delete cgi` a
  l'interieur de la boucle. Le destructeur ~CGI appelait
  CGI::close(int) qui faisait `_cgis.erase(it)` sur le MEME map
  en cours d'iteration. Resultat: ++it sur un iterateur invalide.

Fix:
  - CGI::close(int) devient un simple wrapper de ::close() POSIX,
    plus de mutation du map.
  - Les inserts dans _cgis sont deplaces a la fin de CGI::pipe()
    (apres la validation complete), supprimant le besoin de
    nettoyer les chemins d'erreur.
  - Dans private.cpp, erase explicite aux 3 points de fermeture
    normale cote parent (fork-fail, fin de write, fin de read).

Utilite:
  Elimine le use-after-free qui crashait sous siege. Valgrind
  passe de "Invalid read of size 8" a 0 erreur.


[2] DOUBLE-CLOSE des pipes CGI
-------------------------------
Fichiers: srcs/CGI/private.cpp, srcs/CGI/CGI.cpp

Probleme:
  Apres `close(_pipe[0])` dans _handleReadingState, le fd n'etait
  pas remis a -1. Le destructeur ~CGI re-fermait le fd. Si entre
  les deux closes, accept()/pipe() reallouait le meme numero de
  fd pour autre chose, on fermait le fd d'autre chose.

Fix:
  - Apres chaque close() des pipes dans private.cpp, mettre
    _pipe[0/1] ou _stdinPipe[0/1] a -1.
  - ~CGI ferme maintenant aussi _stdinPipe[0/1] (manquait avant).

Utilite:
  Elimine les erreurs "Bad file descriptor" intermittentes sur
  epoll_ctl et send. Elimine la confusion "recv() on non-socket"
  qui venait d'un fd de pipe recycle en socket client.


[3] EAGAIN interprete comme ERREUR FATALE
------------------------------------------
Fichier: srcs/CGI/private.cpp

Probleme:
  Les pipes CGI sont non-bloquants. Quand read() ou write()
  renvoient -1 avec errno=EAGAIN (pas encore de donnees / buffer
  plein), le code traitait ca comme une erreur et envoyait
  une reponse 500 + tuait le CGI par SIGTERM. Le child Python
  en cours de flush se retrouvait avec BrokenPipe.

Fix:
  - Dans _handleReadingState: sur `bytes < 0`, retour silencieux
    (retentera au prochain EPOLLIN).
  - Dans _handleWritingState: idem sur write() < 0.

Utilite:
  ELIMINE TOTALEMENT LES 500 FANTOMES. Le ratio 200/500 passe de
  50/50 a 100/0. Les BrokenPipe Python diminuent drastiquement.


[4] State machine inconsistante (SENDING_RESPONSE)
---------------------------------------------------
Fichier: srcs/network/Client.cpp

Probleme:
  5 call sites sur 7 appelaient switchToEpollOut() sans avoir mis
  _state a SENDING_RESPONSE. Quand EPOLLOUT tirait ensuite,
  Client::send() etait appelee dans un etat ambigu.

Fix:
  - setResponse() et setErrorResponse() font maintenant
    `_state = SENDING_RESPONSE` a la source.
  - send() a un guard: si state != SENDING_RESPONSE, retourne -1.

Utilite:
  Garantit qu'une reponse n'est envoyee que quand c'est
  reellement attendu. Casse la boucle de "reponses vides" avec
  headers: {} qui saturait les logs.


[5] Plusieurs CGIs concurrents sur un meme client
--------------------------------------------------
Fichier: srcs/network/Server.cpp

Probleme:
  Avec siege en HTTP/1.1 keep-alive, un meme client pouvait
  recevoir une nouvelle requete pendant qu'un CGI etait en cours.
  handleRequest creait alors un SECOND CGI, orphelinant le
  premier. Le parent ne lisait plus son pipe, le child Python
  bloquait puis recevait BrokenPipe.

Fix:
  Guard en tete de handleRequest:
    if (client->getState() != Client::NO_CGI) return;

Utilite:
  Un seul CGI actif par client a la fois. Elimine les orphelins
  et leurs BrokenPipe en cascade.


[6] Client::runCGI utilisait operator[] sur le map
---------------------------------------------------
Fichier: srcs/network/Client.cpp

Probleme:
  `CGI* cgi = _cgis[cgiFd];` inserait un pointeur null si la cle
  n'existait pas (par ex. apres une erase prealable), puis
  `cgi->run()` deferencait null.

Fix:
  Utiliser `_cgis.find(cgiFd)` et sortir proprement si absent.

Utilite:
  Evite un segfault potentiel lors de races sur le state machine.


[7] Log "received request" premature
-------------------------------------
Fichier: srcs/network/Client.cpp + Server.cpp

Probleme:
  Client::read() loggait "received request from client" a chaque
  appel de recv(), meme pour un read partiel. Les logs etaient
  impossibles a lire: 3 "received request" pour 1 vraie requete.

Fix:
  - Extraire le log dans Client::logRequest().
  - L'appeler depuis Server::handleRequest() uniquement apres
    isRequestComplete().

Utilite:
  Les logs deviennent lisibles, ce qui a permis de debugger les
  bugs suivants.


[8] Send/recv EAGAIN sur les SOCKETS clients
---------------------------------------------
Fichier: srcs/network/Client.cpp

Probleme:
  Client::read() et Client::send() traitaient un retour -1
  comme un echec et le caller declenchait _remove(), fermant la
  connexion. Sur un event EPOLLIN/EPOLLOUT spurieux, on tuait
  une connexion saine.

Fix:
  - read(): sur `bytes < 0`, retourne 1 (pas -1). Le caller ne
    declenche pas _remove.
  - send(): idem sur bytes < 0. Sur bytes == 0 (rare), retourne
    -1 (vraie deconnexion).
  - Retrait de l'appel a strerror(errno) apres read/write qui
    violait le subject 42.

Utilite:
  Reduit les fermetures intempestives. Les "Connection reset by
  peer" cote siege diminuent.

NOTE SUR SIEGE
--------------
Par defaut, siege envoie "Connection: close" pour chaque requete,
meme avec -H "Connection: keep-alive" ou configuration .siegerc
(siege ne respecte pas toujours ces options). Le serveur honore
correctement ce header et ferme la connexion apres la reponse.

Les "Connection reset by peer" vus cote siege viennent du close()
abrupt du serveur combine au pattern d'ouverture/fermeture rapide
de siege. Un serveur de prod utiliserait shutdown(SHUT_WR) avant
close pour une fermeture gracieuse, mais ce n'est pas requis pour
webserv.
