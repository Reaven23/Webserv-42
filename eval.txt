[OK] Même port déclaré plusieurs fois → rejeté au parsing
     Cas 1 : deux server blocks avec listen 9092
       → "Parsing error: duplicate listen 0.0.0.0:9092"
     Cas 2 : un seul server block avec listen 9092; listen 9092;
       → "Parsing error: duplicate listen 0.0.0.0:9092"
     Implémentation : srcs/config/Config.cpp:56 ("duplicate listen") via
     std::set<std::string> uniqueListens dans Config::validate().
     Le binaire retourne avec un message d'erreur clair avant d'appeler listen().

[OK] Deux instances webserv qui partagent un port → la 2ᵉ échoue proprement
     Démo :
       ./webserv conf1 &  (listen 9093)           → server 1 démarre
       ./webserv conf2 &  (listen 9093)           → server 2 échoue avec
                                                      "bind(): Fatal error:
                                                       Address already in use"
                                                      et le process exit.
       curl :9093                                 → server 1 répond toujours 200 OK.
     Comportement attendu : c'est l'OS (SO_REUSEADDR désactivé ou bind collision)
     qui refuse le second bind.

[..] Subtilité config : une directive = une ligne
     Le parser ne split pas sur ';', il lit ligne par ligne (srcs/config/Config.cpp
     parseLocationBlock, parseServerBlock). Conséquence :

       location / { methods GET; root /tmp/x; }   ← TOUT sur une ligne
       → la directive "methods" absorbe "GET; root /tmp/x;" et la location n'a
         plus aucune méthode valide → toutes les requêtes reçoivent 405 avec un
         header Allow bizarre ("Allow: GET;, root, /tmp/x").

     Pour que la config marche il faut écrire chaque directive sur sa propre ligne :
       location / {
           methods GET;
           root /tmp/x;
       }

     Pas un bug, mais le correcteur qui écrit vite peut se planter et croire que
     ça ne marche pas. À mentionner en soutenance.
